# 基于新闻舆情的多模态股票预测系统设计

本文档是本项目内的可实施版本，目标不是描述一个“理想中的量化平台”，而是明确在当前代码仓库里，如何把新闻舆情能力落成可训练、可验证、可上线的功能。

## 1. 目标

### 1.1 业务目标

在现有股票预测系统基础上，引入新闻舆情特征，验证新闻是否能为 A 股日频预测带来稳定增量价值。

### 1.2 一期目标

一期只解决一个清晰问题：

- 输入：截至交易日 `t` 收盘可见的新闻、技术指标、静态公司/行业关联信息
- 输出：股票在未来 `5` 个交易日内上涨的概率
- 标签定义：

```text
y_t = 1, if close[t+5] / close[t] - 1 > 0.01
      0, otherwise
```

说明：

- `1%` 是一期默认阈值，用于过滤纯噪声波动
- 若后续验证发现样本分布失衡，可改为 `0%` 或分位数标签，但必须单独评估

### 1.3 非目标

一期不做以下内容：

- 分钟级或 tick 级预测
- 纯文本大模型端到端交易
- 自动下单
- 复杂知识图谱推理
- “保证盈利”的策略承诺

## 2. 与当前仓库的关系

当前仓库已经具备新闻与多模态雏形，但还不够严谨，不能直接视为可验证版本。

### 2.1 已有模块

| 模块 | 现状 | 文件 |

| 行情数据获取 | 已有 | `src/data_source_manager.py` |
| 新闻抓取 | 已有 | `src/news_crawler.py` |

| 相关性图谱 | 已有静态版 | `src/relevance_graph.py` |

| Web API | 已有接口 | `app.py` |

### 2.2 当前主要问题

当前实现距离“可实施”还差这几步：

1. 训练目标和标签未固定，模型输出与回测目标并不严格一致。
2. 新闻与价格的时间对齐不严谨，存在未来信息泄漏风险。
3. 训练脚本把一份新闻列表重复用于多个历史窗口，不符合真实时序。
4. 当前验证方式偏弱，必须引入滚动时间切分而不是随机切分。
5. 当前多模态模型偏重架构展示，轻于数据质量和样本定义。

## 3. 一期范围

### 3.1 输入范围

一期只使用以下三类特征：

1. 新闻聚合特征
2. 技术面特征
3. 静态关联特征

### 3.2 输出范围

一期只输出以下结果：

1. `up_probability`: 未来 5 日上涨概率
2. `expected_return`: 基于概率映射的预期收益
3. `confidence`: 置信度
4. `feature_summary`: 用于页面展示的解释性摘要

### 3.3 训练范围

一期训练以“日频截面样本”为主，不做原始全文 BERT 编码。

原因：

- 当前项目最缺的是时序一致的数据集，而不是更复杂的模型
- 聚合特征更容易做回测、排错和解释
- 先验证新闻是否有增量，再决定是否引入文本编码器

## 4. 数据定义

### 4.1 样本单位

一期样本单位定义为：

```text
sample = (stock_code, trade_date)
```

每一行样本都表示：

- 在 `trade_date` 收盘后做预测
- 只能使用 `trade_date` 收盘前可见的数据
- 去预测 `trade_date + 5` 的标签

### 4.2 时间对齐规则

这是一期最重要的规则。

#### 日频样本时间窗

对于交易日 `t`：

- 行情特征：只使用 `<= t` 的 K 线数据
- 新闻特征窗口：`(prev_trading_day 15:00, t 15:00]`
- `t` 日 `15:00` 之后发布的新闻，归入下一交易日样本

#### 数据泄漏禁止项

以下做法禁止进入正式训练数据：

1. 使用 `t+1` 之后的新闻去构造 `t` 的样本
2. 用未来收益反推新闻情感作为训练特征
3. 随机切分时间序列样本后直接训练并汇报结果
4. 对整段数据先标准化再切训练集和测试集

## 5. 特征方案

### 5.1 新闻特征

一期不直接喂原始文本，先聚合为结构化特征。

建议使用：

- 新闻数量：`news_count`
- 情感均值：`avg_sentiment`
- 时间加权情感：`weighted_sentiment`
- 重要性：`avg_importance`, `max_importance`
- 情感分布：`positive_ratio`, `negative_ratio`
- 来源分散度：`source_count`
- 行业影响向量：`sector_bank`, `sector_consumer`, ...
- 事件标签：`event_policy`, `event_earnings`, `event_mna`, ...

### 5.2 技术面特征

沿用现有项目中已经可计算的技术指标：

- `ret_1d`, `ret_5d`, `ret_20d`
- `vol_5d`, `vol_20d`
- `rsi`
- `macd`
- `bb_position`
- `ma5_gt_ma10`
- `ma10_gt_ma20`
- `volume_ratio`

### 5.3 静态关联特征

一期只使用静态和弱动态信息：

- 公司所属行业
- 主题概念标签
- 股票相关性矩阵摘要
- 行业内相对位置

相关信息优先复用：

- `src/relevance_graph.py`
- `src/company_info_engine.py`

## 6. 模型策略

### 6.1 一期模型优先级

按下面顺序实现，不要一开始就上重模型。

1. `Logistic Regression`
2. `LightGBM / XGBoost`
3. `MLP / Dense multimodal`
4. 文本向量模型

### 6.2 推荐一期基线

推荐一期正式基线：

- 训练数据：`model_dataset.parquet`
- 模型：`LightGBMClassifier`
- 标签：`label_up_5d`
- 输出：上涨概率

原因：

- 训练快
- 特征重要性可解释
- 对结构化特征更合适
- 比当前的“先做神经网络再补数据集”更稳

### 6.3 多模态神经网络进入条件

只有在以下条件满足时，才进入第二阶段：

1. 新闻聚合特征相对纯技术面存在稳定增量
2. 滚动回测在多个时间段都成立
3. 数据集和时间对齐规则已经固定

## 7. 训练与验证

### 7.1 数据切分

严格使用时间切分。

推荐：

- 训练集：最早 70%
- 验证集：中间 15%
- 测试集：最后 15%

正式评估再补一层滚动验证：

```text
train: 2022-01 ~ 2023-06
valid: 2023-07 ~ 2023-09
test : 2023-10 ~ 2023-12

然后向前滚动窗口重复
```

### 7.2 评估指标

至少同时看三类指标：

#### 分类指标

- `AUC`
- `F1`
- `Precision@TopK`

#### 收益指标

- Top-N 平均未来收益
- 胜率
- 最大回撤
- 超额收益

#### 稳定性指标

- 月度表现分布
- 行业分组表现
- 高波动期间表现

### 7.3 通过标准

一期是否继续推进，以以下标准决定：

1. 新闻 + 技术面模型优于纯技术面模型
2. 提升不只出现在单一时间窗口
3. 回测结果在考虑交易成本后仍为正增量

## 8. 回测规则

### 8.1 策略规则

一期使用最简单且可解释的策略：

- 每个交易日收盘后打分
- 选择概率最高的 `Top N` 股票
- 持有 `5` 个交易日
- 等权或按概率加权

### 8.2 成本假设

回测必须显式纳入成本：

- 手续费：`0.03%`
- 卖出印花税：按当时市场规则配置
- 滑点：建议先用 `0.02%`

### 8.3 回测输出

至少输出：

- 收益曲线
- 回撤曲线
- 换手率
- 行业分布
- 每日持仓明细

## 11. 实施路线图

### Phase 1: 数据正确性

目标：先把样本定义做对。

任务：

1. 固定标签定义和时间对齐规则
2. 产出 `news_raw.parquet`
3. 产出 `market_daily.parquet`
4. 产出 `news_daily_features.parquet`

验收：

- 能按 `stock_code + trade_date` 回放任一条样本
- 能证明样本只使用当时可见信息

### Phase 2: 基线验证

目标：验证新闻是否有增量。

任务：

1. 构建 `model_dataset.parquet`
2. 训练纯技术面基线
3. 训练新闻 + 技术面基线
4. 输出离线评估报告

验收：

- 有统一指标表
- 能比较两类模型差异

### Phase 3: 回测闭环

目标：把分类效果转换为策略效果。

任务：

1. 实现截面选股回测
2. 纳入交易成本
3. 输出收益、回撤、换手等指标

验收：

- 至少一个滚动时间窗口中有稳定正增量

### Phase 4: 多模态增强

目标：在基线成立后再上更复杂模型。

任务：

1. 改造 `src/multimodal_model.py`
2. 引入文本向量或轻量文本编码器
3. 做与基线模型的增量对比

验收：

- 新模型必须优于基线，否则不替换

## 12. 风险与约束

### 12.1 技术风险

- 新闻噪声高
- 标题党影响情感判断
- 新闻发布时间不可靠
- 概念映射容易主观化

### 12.2 合规与数据风险

- 新闻抓取需遵守来源站点规则
- 缓存内容注意版权与再分发边界
- 生产环境优先使用授权数据源

### 12.3 建模风险

- 市场对公开新闻反应很快
- 新闻增量可能只在少数事件期成立
- 容易过拟合于热点板块和极端时期

## 13. 最终决策原则

后续开发遵循以下顺序：

1. 先保证数据时间正确
2. 再验证新闻特征是否有增量
3. 最后才增加模型复杂度

判断一个版本是否值得保留，不看“模型看起来多先进”，只看三件事：

1. 是否无泄漏
2. 是否可复现
3. 是否在滚动回测中优于基线

## 14. 当前版本结论

这个方向值得做，但实施顺序必须调整。

当前仓库已经具备新闻抓取、影响分析、相关性图谱和多模态预测原型，下一步不应继续堆模型结构，而应优先完成：

1. 标准化日频数据集
2. 严格时间对齐
3. 结构化基线模型
4. 滚动评估与截面回测

等这四步跑通后，再考虑引入更重的文本编码、多模态融合和更复杂的图谱特征。
